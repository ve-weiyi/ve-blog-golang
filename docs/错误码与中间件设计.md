# 错误码与中间件设计

## 一、错误码设计

### 设计原则

按**错误场景**而非业务模块分类。同一错误码可跨模块复用（如 `430 资源不存在` 适用于用户、文章、评论等所有资源）。

```
格式：3 位数字
百位 = 错误场景大类
十位/个位 = 场景内子分类
```

### 完整码表

```
2xx — 成功
├── 200  成功

3xx — 流控/安全
├── 300  请求过于频繁                  (AntiReply 时间戳过期)
├── 301  请求被限流                    (RateLimit 超配额)
├── 302  验证码校验失败                (图形验证码/短信验证码)
└── 303  请求签名无效                  (DeviceToken 签名错误、防重放)

4xx — 客户端错误
│
├── 400-409  请求参数
│   ├── 400  参数错误                  (通用，兜底)
│   ├── 401  参数缺失                  (必填 Header/Query/Body 缺失)
│   ├── 402  参数格式错误              (邮箱格式、手机号格式等)
│   └── 403  参数值不允许              (枚举值非法、范围越界等)
│
├── 410-419  身份认证
│   ├── 410  未登录                    (Token 缺失或无效)
│   ├── 411  登录已过期                (Token 过期)
│   ├── 412  密码错误                  (登录密码不匹配)
│   ├── 413  账号已被禁用              (用户状态为禁用)
│   └── 414  验证码错误                (短信/邮箱验证码)
│
├── 420-429  权限授权
│   ├── 420  无操作权限                (RBAC 权限不足)
│   └── 421  角色不匹配                (需要特定角色)
│
└── 430-449  业务规则
    ├── 430  资源不存在                (通用：用户/文章/评论等不存在)
    ├── 431  资源已存在                (创建时冲突：邮箱已注册等)
    ├── 432  资源状态不允许当前操作     (文章已发布不可删除等)
    └── 433  操作不被允许              (不能给自己降权等)

5xx — 服务端错误
├── 500  服务器内部错误               (未知错误，兜底)
├── 501  数据库操作失败               (SQL 执行异常)
├── 502  外部服务调用失败             (OSS/SMS/第三方 API)
└── 503  服务超时                     (下游服务超时)
```

### 使用方式

```go
import "github.com/ve-weiyi/ve-blog-golang/infra/biz/bizcode"

// 返回错误给客户端
bizerr.NewBizError(bizcode.CodeResourceNotFound, "用户不存在")
bizerr.NewBizError(bizcode.CodeRateLimited, "请求过于频繁")
```

### 代码位置

`infra/biz/bizcode/biz_code.go`

## 二、中间件设计

### 2.1 go-zero 内置中间件

以下中间件由 go-zero 框架自动挂载，全默认开启，无需手动配置。

| 中间件 | 默认 | 配置项 | 说明 |
|---|---|---|---|
| Trace | on | - | OpenTelemetry 链路追踪，traceId 贯穿 API → RPC |
| Log | on | - | 请求日志（method/path/status/duration） |
| Prometheus | on | - | 暴露 `/metrics` 端点供 Prometheus 采集 |
| MaxConns | on | `MaxConns: 10000` | 最大并发连接数 |
| Breaker | on | - | 接口级熔断保护 |
| Shedding | on | `CpuThreshold: 900` | CPU 超过 90% 时自动降载 |
| Timeout | on | `Timeout: 3000` | 请求超时 3 秒 |
| Recover | on | - | panic 恢复，避免进程崩溃 |
| Metrics | on | - | 接口耗时统计 |
| MaxBytes | on | `MaxBytes: 1048576` | 请求体大小限制 1MB |
| Gunzip | on | - | 自动解压 gzip 请求体 |

内置链执行顺序：

```
Trace → Log → Prometheus → MaxConns → Breaker → Shedding
  → Timeout → Recover → Metrics → MaxBytes → Gunzip
  → [自定义中间件链]
```

### 2.2 自定义中间件

#### 全局链（blog-api / admin-api 共用）

```
RateLimit → CtxMeta → AntiReply → DeviceToken
```

| 顺序 | 中间件 | 文件 | 职责 |
|---|---|---|---|
| 1 | RateLimitMiddleware | `infra/middlewarex/ratelimit_middleware.go` | 基于 IP+Path 的滑动窗口限流 |
| 2 | CtxMetaMiddleware | `infra/middlewarex/meta_middleware.go` | 将 HTTP Header 写入 context metadata，透传 gRPC |
| 3 | AntiReplyMiddleware | `infra/middlewarex/antireply_middleware.go` | 防重放：校验 timestamp 未超过 3600 秒 |
| 4 | DeviceTokenMiddleware | `infra/middlewarex/terminaltoken_middleware.go` | 游客签名校验：`token = md5(terminalId, timestamp)` |

设计要点：

- **RateLimit 在最前**：在耗 CPU 的签名校验之前拦截洪水流量
- **默认放行**：AntiReply 和 DeviceToken 在缺少相应 Header 时放行，兼容无客户端签名的普通请求

#### admin-api 路由组链

```
Public（登录/验证码等）:    无额外中间件
需登录（个人中心等）:       AdminAuth
需权限（CRUD 等）:          AdminAuth → Permission → OperationLog
```

| 中间件 | 文件 | 职责 |
|---|---|---|
| AdminAuthMiddleware | `service/admin/api/internal/middleware/adminauth_middleware.go` | JWT Token 校验，区分过期/无效 |
| PermissionMiddleware | `service/admin/api/internal/middleware/permission_middleware.go` | RBAC 权限校验（API + 菜单） |
| OperationLogMiddleware | `service/admin/api/internal/middleware/operationlog_middleware.go` | 审计日志：记录操作人、接口、请求/响应内容 |

#### blog-api 路由组链

| 中间件 | 文件 | 职责 |
|---|---|---|
| UserAuthMiddleware | `service/app/api/internal/middleware/userauth_middleware.go` | 用户 JWT Token 校验 |

### 2.3 RateLimit 限流策略

| 服务 | 窗口 | 配额 | Redis Key 前缀 | 含义 |
|---|---|---|---|---|
| blog-api | 1s | 50 | `blog:ratelimit:global:` | 每 IP 每接口 50 QPS |
| admin-api | 1s | 100 | `blog:ratelimit:admin:` | 每 IP 每接口 100 QPS |

实现基于 Redis 滑动窗口（Lua 脚本），使用 go-redis `*redis.Client` 直接执行，复用业务已有的 Redis 连接。脚本位于 `infra/middlewarex/period_script.lua`。

扩展示例——对登录接口加更严格限制：

```go
// 在 routes.go 中对特定路由组挂载
loginLimiter := middlewarex.NewPeriodLimit(60, 5, rds, "blog:ratelimit:login:")
// 每分钟每 IP 最多 5 次
```

### 2.4 中间件间错误码使用对照

| 中间件 | 场景 | 使用码 |
|---|---|---|
| RateLimitMiddleware | 超出配额 | `CodeRateLimited (301)` |
| AntiReplyMiddleware | 时间戳过期 | `CodeRequestSignInvalid (303)` |
| DeviceTokenMiddleware | Header 缺失 | `CodeParamMissing (401)` |
| DeviceTokenMiddleware | 签名校验失败 | `CodeRequestSignInvalid (303)` |
| UserAuthMiddleware | uid/token 缺失或无效 | `CodeUnauthenticated (410)` |
| UserAuthMiddleware | Token 过期 | `CodeLoginExpired (411)` |
| AdminAuthMiddleware | uid/token 缺失或无效 | `CodeUnauthenticated (410)` |
| AdminAuthMiddleware | Token 过期 | `CodeLoginExpired (411)` |
| PermissionMiddleware | uid 缺失 | `CodeUnauthenticated (410)` |
| PermissionMiddleware | 权限不足 | `CodeNoPermission (420)` |
